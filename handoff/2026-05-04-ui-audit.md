## § 1. Workflow inventory

## § 2. Per-workflow assessment

Severity legend: 🔴 blocking / data-loss risk · 🟡 friction / confusion · 🟢 nit / polish.
File:line refs are absolute paths inside the worktree.

### Workflow 1 — Discord login 🟢

**File:** `app/views/sessions/new.html.erb`

- 🟢 **First impression is plain.** Single GB-themed card on a `min-height: 80vh` flex container. The `&#9876;` (hammer-and-pick) glyph at `sessions/new.html.erb:3` carries the visual weight; below it is uppercase "SOUL LINK" + "POKEMON PLATINUM" + a Discord SVG button. Functional but doesn't do any work to set the run-tracker tone.
- 🟢 **No error / failed-auth state.** `sessions_controller.rb` has a `failure` action (route `/auth/failure`); no template for it. A user kicked back from OAuth will see the bare flash. Acceptable but invisible.
- 🟢 **Footer disclaimer is 9px on `var(--d2)`** — `sessions/new.html.erb:16` — borderline contrast on the light card.

### Workflow 2 — Run management (`/runs` page) 🟡

**File:** `app/views/runs/index.html.erb`

- 🔴 **`End Current Run` has no confirmation** — `runs/index.html.erb:44-47`. Single click ends the active run. Compare to MARK DEAD which has a full confirm modal. Asymmetric handling of destructive actions.
- 🟡 **Five buttons in one flex row** — line 43: `End Current Run`, `Setup Discord`, `Generate Emulator ROMs`, `Regenerate ROMs`, plus a status pill. With `display: flex; gap: 8px;` and no wrap fallback, they overflow on narrow viewports.
- 🟡 **Past-run cards are at `opacity: 0.6`** — line 92 — combined with body-text already at `var(--d2)` over the card's own `var(--l1)`, the contrast is poor. They are also non-interactive (no `cursor: pointer`, no link wrapper) — compare to the dashboard runs tab where they ARE links.
- 🟡 **`gb-grid-4`** is hard-coded at four columns regardless of viewport — `pixeldex.css` defines `gb-grid-4` without a media query. Below ~360px the four stat tiles squeeze.
- 🟢 No empty state for "you've never had a run." Plain "No previous runs." card.

### Workflow 3 — Run management (RUNS tab inside dashboard) 🔴

**File:** `app/views/dashboard/_runs_content.html.erb`

- 🔴 **Same lack of confirmation on END RUN** — `_runs_content.html.erb:73-76`.
- 🔴 **The dashboard tab and `/runs` page render almost the same panel with different button sets** — the dashboard tab has `VIEW`, `END RUN`, `SETUP DISCORD`; the `/runs` page additionally exposes `Generate Emulator ROMs` and `Regenerate ROMs`. Same controller (`run-management`), same DOM targets, divergent buttons. Users reasonably expect the two screens to be the same. Today they are not.
- 🟡 **History cards use `cursor: pointer`** but the wrapper uses `text-decoration: none; color: inherit;` — `_runs_content.html.erb:106` — link affordance is invisible. Border-width changes on the current run (line 107) but it's subtle.
- 🟡 **VIEWING / ACTIVE / 🏆 COMPLETE pills crowd the right side** — line 26-33. Order is "🏆 COMPLETE" then "ACTIVE" but a complete run shouldn't also be active; this combination is impossible. Dead code inviting confusion.

### Workflow 4 — Save-slot ingestion + emulator 🔴

**Files:** `app/views/emulator/show.html.erb`, `_save_slots_sidebar.html.erb`, `_run_sidebar.html.erb`, `_run_sidebar_card.html.erb`

- 🔴 **DELETE has no confirmation** — `_save_slots_sidebar.html.erb:149`. Single click destroys a save slot. There's no undo. The delete button is in a flex-wrap row alongside Download / Make Active, all 9px text — the smallest button on the row is the most destructive.
- 🔴 **Six different "not ready" panels** in `emulator/show.html.erb:15-58` — NO ACTIVE RUN, ROMS NOT GENERATED YET, NO ROM AVAILABLE, ROM GENERATING…, ROM GENERATION FAILED, plus the silent "still mid-claim" race window. All use the same `gb-card` wrapper; nothing differentiates "you need to do X" from "the system is busy" from "something went wrong." User has to read the 11px body copy to figure out which.
- 🔴 **Overwrite-pending mode is a hidden overlay revealed by clicking a slot when full.** No toast, no banner, no header-state change indicating "you're now in overwrite mode." `_save_slots_sidebar.html.erb` has the markup but it's spring-loaded behind an interaction the user didn't ask for explicitly.
- 🟡 **`Make Active` button is hidden when active, leaving an asymmetric row** — line 130-145 area. `Download | Make Active | Delete` becomes `Download | Delete` on the active slot. Small visual jolt.
- 🟡 **`Clear All Slots` is rendered next to the slot list with no extra friction.** It has no destructive-action treatment beyond its `gb-btn-danger` class (`_save_slots_sidebar.html.erb:161`).
- 🟡 **Roster cards stack 8–10 lines of 9–10px text** — `_run_sidebar_card.html.erb` — Trainer name, play time, money, map, badges, TID/SID, Pokédex caught/seen, HOF, TID conflict, seed. Wall of text. No visual hierarchy beyond the status pill at top.
- 🟡 **`peso sign &#8369;` for money** — `_run_sidebar_card.html.erb` ~line 75 — odd choice for Pokédollars. The series uses ₽ (`U+20BD`) or just `$`. Genre-correct would be `P̄`-style or just no symbol.
- 🟢 **Seed display is 9px, opacity 0.7, `word-break: break-all`** — visually noisy, not selectable-friendly, and useful enough during debugging that a click-to-copy would help.

### Workflow 5 — Catch logging (manual NEW CATCH modal) 🟡

**File:** `app/views/dashboard/_catch_modal.html.erb`

- 🟡 **The modal is nicknamed-first.** Field order is NICKNAME → LOCATION → SPECIES. The natural mental order in a Nuzlocke is "I caught X at Y, let me name it." Presenting nickname first forces the user to commit to a name before recording what they actually did.
- 🟡 **Three required fields, no progressive enclosure.** All three inputs are required at once with no stepwise guidance. If the user opens the modal before they've decided a nickname, they're stuck.
- 🟡 **"Catch!" button is mixed-case** while every other primary button in the app is uppercase. `_catch_modal.html.erb:62`.
- 🟡 **Modal is missing `role="dialog"` + `aria-modal="true"` + `aria-labelledby`** — only the close button has an `aria-label`. Same pattern in every other modal in the app (`_pokemon_modal.html.erb`, `_mark_dead_modal.html.erb`, `_reset_draft_modal.html.erb`).
- 🟢 No focus trap. After-close focus restoration relies on browser default.

### Workflow 6 — Catch logging (AUTO-DETECTED CATCHES) 🟡

**File:** `app/views/dashboard/_pc_box_content.html.erb` (lines 17–82)

- 🟡 **Four badges, four colors, no legend.** `1ST` and `TRADE-IN` use `var(--amber)`; `EVENT` and `OFF-FEED` use `var(--d2)`. The amber-vs-green grouping has no semantic basis — `1ST` is a positive ("first encounter, log this"), `TRADE-IN` is a warning ("not a real catch"). Sharing a color flattens that distinction.
- 🟡 **`<details><summary>STATS</summary>` block** at line 59-74 is good progressive disclosure but the summary is tiny (9px), the chevron is the browser default, and inside the block the rows read `MOVE 1: #58 · PP 30 · ↑0` — raw move ID instead of the move name we already lookup at the parser level (KG-17 / Step 19). The decoded move name is on `auto_catches[i].moves[i]["id"]`, not `["name"]`, so this view never sees it. **Action item for future ship: thread the move name through.**
- 🟡 **OFF-FEED used to render unstyled** — `var(--d3)` was undefined. Fixed inline in this audit (commit `7618924`).
- 🟢 **Empty-grid padding cells** — lines 78-80, 105-107, 131-135, 159-163 — are present in DOM with `class="box-cell empty"`. Visual padding to keep the grid rectangular; small accessibility nit (empty `<div>`s are noise to screen readers).

### Workflow 7 — Pokémon detail / mark dead 🟡

**Files:** `_pokemon_modal.html.erb`, `_mark_dead_modal.html.erb`

- 🟡 **Modal is 520px wide and tall.** `_pokemon_modal.html.erb:9`. On a 1024-wide screen with the existing PC layout this fits; on a phone it scrolls inside its viewport with no visual scroll affordance.
- 🟡 **`MARK DEAD` and `SAVE` are siblings on the same row** — `_pokemon_modal.html.erb:104-118`. The destructive action is one tap away from the primary save action. Adjacency makes finger-fat errors easy. The MARK DEAD button is `gb-btn-danger` (good) but should be visually segregated, not placed in the same flex row.
- 🟡 **Linked Pokémon section is JS-populated and unindicated when empty** — line 88-92. If JS fails or the group has no linked Pokémon, the section header sits over a blank div.
- 🟢 **GROUP NICKNAME label in the same modal as individual species edit** is confusing — naming the group lives at line 95-101, embedded in a "this Pokémon's details" form. Two scopes (group + member) in one modal.
- 🔴 **Mark-dead modal warning paragraph is dense** — `_mark_dead_modal.html.erb:34` — single paragraph saying "permanently marks every linked pokemon … irreversible." Strong copy; weak typography. The most consequential warning in the app reads like a footer disclaimer.

### Workflow 8 — Pixeldex (species assignment) 🟡

**File:** `app/views/species_assignments/show.html.erb`

- 🟡 **2-fr/1-fr grid never reflows.** Line 47 region — `grid-template-columns: 2fr 1fr;`. On mobile the right column is unusably narrow. No `@media` breakpoint.
- 🟡 **Edit/Delete buttons on every group card with no confirmation on Delete** — `_group_card.html.erb:23-32`. Destroys a group + all member assignments.
- 🟡 **`waiting...` placeholder for non-current-user slots** — `_group_card.html.erb:62-69` — looks identical to the drop-zone for the current user. Non-current-user slots aren't drop targets but visually they suggest you can drag onto them.
- 🟡 **Pokédex tab has search; Pool tab does not** — `species_assignments/show.html.erb:91-101` vs `108-112`. Asymmetric for symmetric tasks.
- 🟢 **Quick-nav buttons** scroll the left column to a specific group but don't visually highlight it on arrival — line 36-42.

### Workflow 9 — Team builder + damage calc 🟡

**Files:** `teams/index.html.erb`, `teams/show.html.erb`, `_pokemon_card.html.erb`, `_quick_calc_modal.html.erb`, `dashboard/_calc_content.html.erb`

- 🟡 **`/teams` index uses `gb-grid-4`** with no mobile breakpoint — `teams/index.html.erb` ~line 9 region. Each player card becomes ~80px wide on a phone.
- 🟡 **The team-builder save-status target sits silently** — `teams/show.html.erb:25` — `data-team-builder-target="saveStatus"` shows `Saving…` / `Saved` / `Error` via JS but starts empty. No "Idle / autosaving on drag" affordance.
- 🟡 **Pool search is on the right column only.** Team column lacks search even though it can hold up to 6 (small) but is the focus of the work.
- 🟡 **Datalists are inconsistent across browsers** — every species autocomplete uses `<datalist>`, which Safari renders differently from Chrome. Consider a custom Stimulus combobox (one already exists in `timeline_controller.js`).
- 🟡 **Damage calc CALC tab and the team-builder Quick Calc modal are two implementations of the same idea.** Two species pickers, two move pickers, two result panels, two stylings. Visit either for reference: `_calc_content.html.erb` and `_quick_calc_modal.html.erb`. Pure duplication.

### Workflow 10 — Map / Route progression 🟡

**File:** `app/views/map/show.html.erb`

- 🟡 **Horizontal timeline has no scroll affordance.** Line 51 sets `overflow-x: auto` and a thin scrollbar; no fade/gradient hint at the right edge, no arrow buttons, no swipe instruction. On desktop, users with non-trackpad mice may not know it scrolls.
- 🟡 **Each location node is 56px wide minimum.** With ~30 nodes for the full game, the track is ~1700px. On a 375px phone, the user has to scroll a lot — and there's no "jump to current segment" shortcut.
- 🟡 **Node labels truncate at 56px max-width with `text-overflow: ellipsis`.** Long route names ("Mt. Coronet — 2F") become "Mt. Coronet …" with no tooltip on touch.
- 🟡 **Slide-out panel is `max-width: 380px`** — line 211. On phones < 380 it covers the whole screen, which is fine; on tablets it covers most of the screen, dimming the timeline behind it. Closing requires hitting the small `&times;` at top-right.
- 🟡 **`Special Encounters` grid is hard-coded `gb-grid-4`** — line 171 — for 4 fixed cells (gift / egg / trade / other). Will fit, but again no responsive fallback.
- 🟢 **Dashboard MAP tab + standalone /map page render different content.** The dashboard renders `_map_content.html.erb` (ASCII map + current location card + recent routes), the standalone `/map` shows the timeline. Two views of "where am I" with no shared interaction.

**Files:** `dashboard/_gyms_content.html.erb`, `gym_ready/show.html.erb`, `gym_drafts/show.html.erb` (audited 2026-05-01), `gym_schedules/index.html.erb`+`show.html.erb`

- 🟡 **`<NEXT` literal text** — `_gyms_content.html.erb:47`: `&lt;NEXT` after the gym leader name. Reads as a stray HTML escape; the author meant a left-arrow + label ("← NEXT"). The combination of `▶ ROARK [ROCK] <NEXT` is visually noisy. (Tempting to fix inline; left for a follow-up because the redesign collapses this row entirely.)
- 🟡 **Defeated gyms render their team snapshot inline** — line 76-90 — as a wall of indented `player_name: species — Lv./ability/nature` lines. With 4 players × 6 groups, this is a paragraph of 9px text inside a card.
- 🟡 **`UNMARK` is only available on the most recently beaten gym** — line 63-72 — and it's a button next to the leader name. Asymmetric — earlier gyms have no edit option.
- 🟡 **`+ ADD TEAM` button** — line 103 — for backfilling which Pokémon fought a gym. Generic label; users won't know what it does without trying.
- 🟢 **Gym Leaders list duplicates between the GYMS tab and the right-sidebar `_map_panel.html.erb`** — same data, two layouts, both visible at the same time on desktop.

## § 3. Cross-cutting issues

These show up in three or more workflows. Worth fixing once, in the design system, rather than per-screen.

### 🔴 Cross-cutting 1 — Destructive actions lack confirmation
- `END RUN` (Workflows 2, 3) — 1-click
- `DELETE` save slot (Workflow 4) — 1-click
- `Clear All Slots` (Workflow 4) — 1-click
- `Delete` group (Workflow 8 — `_group_card.html.erb`) — 1-click
- `Cancel` schedule (Workflow 11) — 1-click

The pattern that DOES work is `MARK DEAD` (Workflow 7) — a separate confirmation modal. Apply that pattern to all five. Either factor a shared `_confirm_modal.html.erb` partial, or use a `data-confirm` Stimulus action.

### 🔴 Cross-cutting 2 — Modals have no `role="dialog"` / `aria-modal` / focus trap
Every modal — `_catch_modal`, `_pokemon_modal`, `_mark_dead_modal`, `_reset_draft_modal`, `_quick_calc_modal`, plus the timeline slide-out panel — is a `<div>` with `position: fixed`. Close buttons have `aria-label` but the modal container itself is unannounced. Tab key escapes the modal into the page behind it.

### 🟡 Cross-cutting 3 — `gb-grid-N` classes hard-code column counts
`gb-grid-2` / `gb-grid-3` / `gb-grid-4` are defined in `pixeldex.css` without media queries. Used in: runs/index, dashboard runs tab, teams/index, gym_ready, map (special encounters), gym_schedules show. Add `@media (max-width: 600px) { .gb-grid-3, .gb-grid-4 { grid-template-columns: repeat(2, 1fr); } }` in one place — it's a 5-line stylesheet change that improves five screens.

### 🟡 Cross-cutting 4 — Tab bars aren't tablists
`dashboard/_tab_bar.html.erb` and `species_assignments/show.html.erb` use `<button>`s with class-based active state. Neither has `role="tablist"` / `role="tab"` / `aria-selected` / `aria-controls`. Screen readers see "row of buttons," not "tab navigation." This is a 6-line ERB change per file.

### 🟡 Cross-cutting 5 — Inline styles everywhere
Almost every view uses `style="..."` for layout (flex / grid / spacing). This makes the visual language hard to evolve — change a card's padding and you need to touch 30 files. The CSS file has `gb-card`, `gb-card-dark`, `gb-grid-N`, `panel-header`, etc., but they cover a fraction of the patterns in use. Recommend extracting the most-repeated inline patterns into utility classes (especially "centered card with padding 16-20", "flex row with space-between", "stat tile") — or, simpler, just inventory what's used most and class-ify.

### 🟡 Cross-cutting 6 — Duplicate views of the same data
- Party (sidebar `_party_panel` + tab `_party_detail` + sometimes strategy)
- PC Box (sidebar `_pc_box_panel` 5-col + tab `_pc_box_content` 3-col)
- Map (sidebar `_map_panel` + tab `_map_content` + standalone `/map`)
- Gym leaders (sidebar `_map_panel` + tab `_gyms_content`)
- Run management (`/runs` page + RUNS tab)
- Damage calc (CALC tab + Quick Calc modal in team builder)

Each duplicate doubles the maintenance cost. Some of this is intentional (sidebar = compact view, main = expanded), but the sidebar/main pairs have drifted into showing different fields. Pick one schema per data type and render it at two sizes.

### 🟢 Cross-cutting 7 — Color-only status indicators
`★` vs `.` for caught vs uncaught (Workflow 6, 11), red vs default for dead (everywhere), amber vs green pills for various states. Add a text label or icon shape alongside color to be colorblind-safe and screen-reader-readable.

### 🟢 Cross-cutting 8 — Hover states are inconsistent
The CSS defines `.gb-nav-link:hover`, `.gb-nav-logout:hover`, `.gb-btn-primary:hover`. Inline-styled buttons (most of the dashboard) get no hover treatment. Cards that are clickable (run history, schedule cards) have `cursor: pointer` but no hover feedback.

### 🟢 Cross-cutting 9 — 9–10px text everywhere
The smallest readable size is `9px` for labels. It is below the threshold most modern apps use (12px). The Press Start 2P font is dense; at 9px on a 2× retina, individual letterforms blur. For non-decorative copy, bump to 11–12px.

## § 4. Recommended redesigns (mocks attached)

Picked by **pain × frequency**. Mockups are in `handoff/`:

| # | Redesign | Mockup file | Why |
|---|----------|-------------|-----|
| **R1** | Dashboard shell + tab navigation | `2026-05-04-ui-audit-mockup-dashboard.html` | Loaded on every visit. Tabs lack ARIA, no mobile reflow, sidebars duplicate main content. Highest frequency × medium pain. |
| **R2** | PC Box / catches surface | `2026-05-04-ui-audit-mockup-pc-box.html` | Densest tab. Auto-detected catches with badge legend, on-team/storage/fallen unification, click affordances, mobile reflow. High pain × high frequency. |
| **R3** | Save-slot management | `2026-05-04-ui-audit-mockup-save-slots.html` | Highest blast radius. Adds confirmation, redesigns the overwrite-pending mode as a foreground decision instead of a hidden overlay, gives roster cards a real hierarchy. High pain × medium frequency. |
| **R4** | Map / Route progression | `2026-05-04-ui-audit-mockup-map.html` | Highest creativity opportunity. Adds scroll affordance, "jump to current" anchor, larger touch targets, an accordion-by-segment fallback for narrow viewports, and a clearer node legend. Medium pain × high frequency. |

I considered run management as a fifth redesign and decided against it — the right move there is an IA decision (collapse `/runs` and the RUNS tab into one canonical surface) more than a visual redesign. That deserves its own conversation, not a mock.

I also considered the gym workflow as a fifth — but it's already partially redesigned in `2026-05-01-gym-draft-redesign.html`, and the remaining surface (gyms tab + gym_ready + gym_schedules) shares the same patterns I'm fixing in R1 and the cross-cutting items.

### What each mockup demonstrates

**R1 — Dashboard shell** (`mockup-dashboard.html`)
- Tab bar restyled as a real tablist with icon + text, sticky on scroll, wraps on narrow viewports.
- Sidebars become a single right-rail "live status" card with tabs (Party / Map / Gyms) instead of three separate panels stacked, freeing 200+ vertical pixels.
- Two-column layout on tablet (sidebar moves to bottom on mobile), three-column on desktop.
- Title bar slimmed down — stats become a single inline row, run selector becomes a pill with a chevron.

**R2 — PC Box** (`mockup-pc-box.html`)
- Auto-detected catches becomes a single "REVIEW PARSED CATCHES" tray at the top, with a one-line legend (`1ST = first encounter on a route` / `TRADE-IN = mark as not real` / etc.).
- Each parsed catch row has a per-row primary action (LOG / SKIP / EDIT) instead of just a badge — the workflow ends at a click, not at staring at the row.
- On-team / storage / fallen unify into one filterable grid with chips (`ON TEAM (4)` / `STORAGE (12)` / `FALLEN (3)` / `ALL`). Filtering is client-side.
- Cells have hover/focus states and obvious click affordance.
- Type-coverage strip moves to the side, no longer a footer.
- Mobile: single-column reflow, larger tap targets, sticky filter chips at top.

**R3 — Save slots** (`mockup-save-slots.html`)
- Slot states are first-class (EMPTY / SAVED / ACTIVE / OVERWRITE-PENDING) with a visible state pill on every slot, not a hidden overlay.
- Delete is behind a confirmation step — the first click swaps the row into a "Confirm delete?" inline state with `Cancel` and `Delete forever`. Same for `Clear All Slots`.
- Overwrite-pending mode is announced by a sticky banner at the top of the column ("Save full — pick a slot to overwrite or [Cancel]"). No more spring-loaded overlays.
- Roster cards re-typeset with a real hierarchy: trainer name (16px) > play time + badges + Pokédex (11px) > the rest collapsible behind a `<details>STATS</details>`.
- Money symbol changed to text "$" (or fully removed — Pokédollars don't have a Unicode glyph that's universally rendered).

**R4 — Map / Route timeline** (`mockup-map.html`)
- Right-edge gradient fade + a "JUMP TO NOW" pill that anchors the scroll on the next-uncaught route.
- Larger node hit targets (72px → 96px) on touch viewports via `pointer: coarse` query.
- Two-row timeline option for mobile: top row = nodes, bottom row = labels — labels can wrap to 2 lines instead of truncating.
- Per-segment collapsible fallback for very narrow viewports — a vertical accordion of segments for phones, where the horizontal timeline is impractical.
- Slide-out panel becomes a sheet that snaps to half-height first; second tap goes full-height. Per-player species inputs use a custom combobox (already exists in `timeline_controller.js`) instead of `<datalist>`.
- Visible legend at the top: ● caught · ☠ dead · ○ uncaught · ★ special · ▣ gym.
