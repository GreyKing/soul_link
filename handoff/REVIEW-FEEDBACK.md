## Should Fix
- `app/views/dashboard/_title_bar.html.erb:81` — Every `<a class="run-option">` has `tabindex="0"`, so a Tab keypress walks through every run option in addition to the trigger pill. The WAI-ARIA listbox pattern Bob declared (`aria-haspopup="listbox"` + `role="listbox"`) usually keeps options out of the Tab order (`tabindex="-1"`) and routes focus into them via the controller's roving-focus / Arrow keys (which Bob already implemented in `run_picker_controller.js`). Recommendation: change the option `tabindex` to `-1` and let the controller's `open()` handler focus the first option as it already does. The Enter / Arrow contract still works without options being tab-stops. Not blocking — keyboard nav still functions; just adds extra Tab stops on every page load.
- `app/views/dashboard/_title_bar.html.erb:94-100` — The `+ START NEW RUN` `<button>` lives inside a `role="listbox"` but does not carry `role="option"`, so a screen reader walking the listbox children skips it. The brief said the menu options activate as `role="option"` (line 35). Recommendation: add `role="option"` to that button (and align with the `tabindex` Should-Fix above). Stays a focusable listbox item; matches the brief's contract.
- `app/javascript/controllers/run_picker_controller.js:81` — `currentIndex = options.indexOf(document.activeElement)` returns `-1` when the trigger (not an option) has focus. The `Home`/`End`/`ArrowDown` paths handle that case correctly (they index into `options[]` directly), but the `ArrowUp` branch with `currentIndex = -1` evaluates `(-1 - 1 + n) % n = n - 2`, jumping the user to the second-to-last option instead of the last. Edge case (only triggers if the user opens the menu via click → trigger still has focus → presses ArrowUp before any Arrow event has moved focus into the menu). Recommendation: when `currentIndex === -1`, ArrowUp should land on the last option and ArrowDown on the first. Under 5 minutes — fix inline or log to BUILD-LOG.
- `app/views/dashboard/_status_rail.html.erb:155-158` — `button_to "START GYM DRAFT →", gym_drafts_path, method: :post, class: "draft-cta", form: { style: "margin: 0;" }` carries an inline style on the wrapper form. The CSS file already has `.dash-r1 .next-battle .draft-cta` rules (lines 2423-2431); the form wrapper is the only thing left with inline style. Audit cross-cutting #5 (inline styles) is what this redesign is moving away from. Recommendation: add `.dash-r1 .next-battle form { margin: 0; }` to `pixeldex.css` and drop the `form: { style: ... }` arg. Cosmetic.
