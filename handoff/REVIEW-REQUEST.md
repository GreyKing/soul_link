## Self-Review

### What would Reviewer most likely flag?

1. **`.rubocop.yml` edit was not in the brief's stated scope.** The brief said: "Diff scope: 1 new migration, 1 model edit, 1 test file edit, 1 schema.rb update, 4 handoff files." I added `.rubocop.yml`. Reason: schema.rb (auto-regenerated) violated `Layout/SpaceInsideArrayLiteralBrackets` after the new index was added. Without an exclude, the project's "rubocop clean" invariant from Step 10 would be lost. The `.rubocop.yml` change is small (4 lines) and fully justified by the schema dumper's behavior. Documented inline. If Reviewer prefers a different fix (e.g., extending the AllCops:Exclude with `inherit_mode: merge: [Exclude]`), I attempted that first — it didn't propagate cleanly from a child config inheriting `rubocop-rails-omakase`. The per-cop exclude is what works.

2. **Backfill safety belt verified locally.** During verification, an early test script's loose `MigrationContext#rollback`/`migrate` calls left dupe data in the dev DB. The migration's backfill check correctly aborted: `Cannot enforce one-active-run-per-guild: guild_id=555555555555555555: 2 active runs. Deactivate the extras manually before re-running, e.g. SoulLinkRun.where(guild_id: <id>, active: true).order(:run_number).limit(<n - 1>).update_all(active: false)`. Cleaned up the test data via `SoulLinkRun.where(guild_id: 555...).destroy_all` and re-ran migrate cleanly.

3. **Two-method `before/after_*_commit` dedup workaround NOT needed here.** Step 9 hit a Rails callback dedup quirk (registering the same method on both `after_create_commit` and `after_update_commit` only kept one). Step 11's `validate :no_other_active_run_for_guild` is a single registration with an `:if` condition — no dedup issue. Verified by the model test passing.

4. **`current(guild_id)` simplification doesn't change behavior on the happy path.** Pre-Step-11: there was at most one active row per guild AS LONG AS the start_run flow was followed; the `order(run_number: :desc).first` was defensive against multiple. Post-Step-11: the DB constraint guarantees at most one. `find_by(guild_id:, active: true)` returns the single row directly. Both query plans use the existing `(guild_id, active)` index.

5. **The "DB-level unique index catches a race" test exercises the actual constraint, not the validation.** It uses `SoulLinkRun.connection.execute` with raw SQL to bypass AR validations. Without the DB constraint, this would silently succeed. With it, it raises `ActiveRecord::RecordNotUnique`. Verified the test passes.

6. **Validation runs on every save where `active?` is true.** That includes routine updates (e.g., `update!(gyms_defeated: ...)` from `mark_beaten` flow). The `where.not(id: id) if persisted?` excludes self, so already-active rows can be updated freely. The "allows updating an already-active run without self-conflict" test pins this.

- **Validation error on duplicate-active create**: `record.errors[:active]` includes "another run is already active for this guild". The Rails controller layer would normally surface this as an `unprocessable_entity` JSON or a flash. The current callers (`RunChannel#start_run`, `discord_bot.rb`, rake tasks) all do `deactivate!` first so they never hit this; the validation is the safety belt for any future caller that might forget.
- **DB-level RecordNotUnique** if a raw-SQL or AR-bypassing path tries to insert a second active row: the `RunChannel` rescue in `start_run` would transmit `{ error: <message> }` to the client; other paths would 500. Acceptable — this is the very thing the constraint is meant to catch.
- **Deploy script behavior**: if production data already has dupe-active runs (expected: it doesn't), `bin/rails db:migrate` aborts with the message naming the offending guild + the cleanup query. Project Owner manually deactivates the extras and re-runs.

## Open Questions / Notes

1. **The deploy migration ran-untested-against-prod.** Production data may or may not have dupe-active rows. The Architect's expectation is that it doesn't (the `start_run` flow always deactivates first), but I haven't queried production to confirm. The migration's safety belt is the verification: if it raises during deploy, that's the signal. Project Owner can also pre-check via rake console: `SoulLinkRun.where(active: true).group(:guild_id).having("COUNT(*) > 1").count`.

2. **The validation message ("another run is already active for this guild") attaches to the `:active` attribute.** Rails surfaces this as `record.errors[:active] = ["another run is already active for this guild"]`. Form-builder UIs would render this on the active checkbox / label. Since `start_run` is a server-side flow with no user form for the active flag, the error would only appear in API/JSON responses or controller flash messages. None of the existing callers display field-specific errors for run, so this is a defense-in-depth signal more than a UX touchpoint.

3. **Test runtime delta**: 8 new tests, no measurable suite-time change (~1.66s wall clock pre and post).

4. **Did NOT add an explicit transaction around `RunChannel#start_run`.** Brief explicitly forbade this. The DB constraint catches the rare race; an explicit transaction is a future polish item.

5. **Did NOT add a `SoulLinkRun.start_for_guild!(guild_id)` helper that wraps deactivate-then-create.** Brief forbade scope creep.
